If I want to run a local instance of strudel, so that I can call functions from another JavaScript file, should I work with the uzu Strudel source code, or another repo?

If you absolutely need to download or fork the full source code (for instance, to host a local version of the entire interactive REPL website or hack on Strudel's core engine), the uzu/strudel repository on Codeberg is the official, up-to-date home of the project. Development migrated there from GitHub.

However, if your primary goal is simply to call Strudel functions from your own separate JavaScript file, you don't need to clone or work inside the source repository at all!

The Better Approach: Use npm Packages
Strudel is built as a highly modular monorepo, and its core engines, parsers, and audio components are published directly to npm under the @strudel namespace. Instead of managing a local instance of the source repository, you can initialize your own project and install only the pieces you need.

Depending on what you want to achieve, you can mix and match these key packages:

@strudel/core: The standalone TidalCycles pattern engine. Use this if you just want to generate, manipulate, and query algorithmic patterns mathematically without necessarily playing audio.  
0xacab

@strudel/webaudio: The engine that handles synths, samples, and audio scheduling via the browser's Web Audio API.  
Yarn Classic

@strudel/web: An opinionated "umbrella" package designed for the browser that bundles the core pattern functions and audio output together without the full REPL user interface.  
strudel.cc

@strudel/mini: The parser for mini-notation string syntax (e.g., "[bd sd, cp*4]").

Quick Example (Via npm)
If you use a modern build tool or bundler (like Vite) in your project, you can pull Strudel primitives right into your own JavaScript code:  
strudel.cc

JavaScript
import { sequence } from '@strudel/core';

// Create a structural pattern entirely in your own JS file
const pattern = sequence('c3', ['eb3', 'g3']);

// Query the events happening between beat 0 and 1
const events = pattern.queryArc(0, 1);
console.log(events);

I might want to use any or many of strudel's features. For what other reasons are npm packages a better approach? I am looking for the easiest method to run strudel locally. Is using the npm packages easier or less prone to issues?

Using the npm packages is significantly easier, faster, and vastly less prone to issues than cloning and building the full source repository.

Here is exactly why working with the npm packages is the superior approach for your goals, along with the easiest method to get up and running locally.

Why NPM Packages Are Less Prone to Issues
1. You Avoid Monorepo Overhead
The main uzu/strudel repository is a massive monorepo managed with pnpm workspaces. It contains the core engine, the entire web-based REPL interface, documentation, tests, and internal build tooling.

The Source Code Route: If you clone it, you have to build the entire ecosystem locally. If a node version mismatch breaks an internal documentation-generation script, your local build fails—even though that script has nothing to do with the music engine you want to use.

The NPM Route: You install only the compiled, production-ready code. You bypass all the build-step landmines of the source repo.

2. Version Locking and Stability
The source repository is a moving target where active development happens. If you clone it, you are running on the bleeding edge. If the maintainers refactor an internal API tomorrow, your custom JavaScript files interacting with it might instantly break. With npm, you pin a specific version (e.g., @strudel/web@1.0.2), ensuring your code never breaks unexpectedly.

3. Separation of Concerns
If you work inside the cloned source code, your custom JavaScript logic becomes tangled with Strudel’s source files. Upgrading Strudel down the line means dealing with painful git merge conflicts. Using npm keeps your code in its own dedicated project repository, completely isolated from Strudel's internals.

The Easiest Method to Run Strudel Locally
If you want a local environment where you can write custom JavaScript to call Strudel functions, the absolute easiest path is to spin up a minimal project using Vite and install the @strudel/web package (which bundles the pattern engine and Web Audio output together).

Here is the complete setup in three steps:

Step 1: Initialize a standard Vite project
Run these commands in your terminal to set up a clean, lightweight local development environment:

Bash
npm create vite@latest my-strudel-app -- --template vanilla
cd my-strudel-app
npm install
Step 2: Install the Strudel web package
Bash
npm install @strudel/web
Step 3: Call Strudel from your JavaScript file
Open the project in your editor, clear out main.js, and import what you need directly:

JavaScript
import { sound, sequence } from '@strudel/web';

// Initialize a simple sequence using Strudel primitives
const tune = sequence('c3', 'e3', 'g3', 'b3').s('saw');

// Create an interactive trigger in your own custom JS/HTML
const button = document.createElement('button');
button.innerText = 'Play Strudel Pattern';
button.onclick = () => {
  // Start the audio context and play the pattern
  tune.play();
};

document.body.appendChild(button);
To run it locally, just type npm run dev. Vite will spin up a local server (usually at localhost:5173) with instant hot-reloading.

The Only Reason to Ever Clone the Source Repo
You should only clone the uzu/strudel source code if your goal is to change how Strudel itself works—for example, if you want to modify its mini-notation parser, rewrite its core scheduling algorithms, or contribute a pull request to the official project.

If you just want to build something with Strudel, stick to npm. It keeps your build chain lightweight and your custom code completely decoupled.

4. The Structured Way: Strudel Kit
If you want an environment specifically pre-optimized for building standalone Strudel projects, the core development team maintains an official scaffolding tool called Strudel Kit (which runs on Astro and Vite under the hood).

Instead of building your own environment from scratch, you can scaffold their official sandbox project:

Bash
npx degit strudel-science/strudel-kit my-strudel-app
cd my-strudel-app
npm install
npm run dev
Why it’s better: It is tailored explicitly for the framework. It comes pre-configured to handle Strudel’s specific audio routing, web workers, and asset paths right out of the box, saving you from troubleshooting audio-context edge cases later on.

What is the best method and approach to use if I want to add a visual dashboard in the future that shows what certain variables in strudel are set to at a given moment, like chord choices, playback speed, gain, and other effects?

I am fine if the display only updates after user input, or once a second, whichever is slower. I do not need it to update too fast.

That makes things substantially easier and removes a ton of engineering overhead. Dropping the requirement for a high-frequency (60fps) refresh loop means you completely bypass the need for complex rendering performance optimization.

Instead of an intensive animation loop, your architecture shifts to a simple Throttled / Event-Driven model. The CPU cost becomes negligible, leaving all the processing power for Strudel's audio engine and your MIDI inputs.

The Simplified Architecture
With this relaxed update rate, you can handle the two directions of data flow using two simple, low-impact mechanisms:

User Input / MIDI to Strudel (Instant & Event-Driven): When you turn a MIDI knob or click a UI button, you want the audio engine to react instantly. Because human inputs happen relatively slowly compared to audio clock cycles, you can update the internal state and the DOM text element directly inside the input's event listener.

Strudel to UI Dashboard (Throttled Poll): If a running Strudel pattern is constantly modulating an effect over time, you don't need to capture every tiny micro-change. You can use a standard setInterval to take a snapshot of Strudel's active parameters exactly once a second (1000 ms) and push that to the screen.

How It Looks in Code (Vanilla JS + Vite)
Because your rendering needs are light, you don't even necessarily need a framework like Svelte anymore—plain vanilla JavaScript inside a Vite project will handle this perfectly and keep your project footprint tiny.

Here is how you can set up this low-frequency dashboard:

JavaScript
import { sequence, initStrudel } from '@strudel/web';

// 1. Storage for our live variables
const state = {
  gain: 0.8,
  speed: 1.0,
  currentNote: 'None'
};

// 2. Outbound: Strudel reads dynamically from our state object
const pattern = sequence('c3', 'e3', 'g3', 'b3')
  .gain(() => state.gain)
  .speed(() => state.speed)
  .onTrigger((hap) => {
    // Continually update our local state cache behind the scenes
    state.currentNote = hap.getParam('note') || 'None';
  });

// 3. Inbound: Handle MIDI or UI adjustments instantly on user interaction
function updateGain(newValue) {
  state.gain = newValue;
  // Update the UI element directly, only when the user changes it
  document.getElementById('gain-display').innerText = `Gain: ${newValue.toFixed(2)}`;
}

// Mocking a MIDI or Slider input event
document.getElementById('gain-slider').addEventListener('input', (e) => {
  updateGain(parseFloat(e.target.value));
});

// 4. Dashboard Readout: Poll the state once per second for automated changes
setInterval(() => {
  // This updates the screen on a strict 1-second heartbeat
  document.getElementById('note-display').innerText = `Playing Note: ${state.currentNote}`;
}, 1000);
Why Vite is Still the Best Fit Here
Even with a highly simplified vanilla JavaScript approach, sticking with a basic Vite template is the right move for a few reasons:

Instant Reloading: As you write code to map your specific MIDI controller's CC values to Strudel functions, Vite will hot-reload your changes in the browser instantly without requiring you to refresh the page or restart a local server.

Asset Management: If you want to load custom audio samples into Strudel locally later on, Vite makes serving local audio files out of a public/ directory completely seamless.

Zero Bloat: A vanilla Vite setup contains almost no boilerplate—just an index.html, a main.js, and a CSS file.

This approach keeps your development environment clean, completely predictable, and highly performant.
