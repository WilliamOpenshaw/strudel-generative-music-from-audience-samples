Are there any downsides to using two ble midi controllers?

Using two Bluetooth Low Energy (BLE) MIDI controllers simultaneously is completely supported by modern operating systems and Strudel, but managing multiple wireless connections over Bluetooth introduces a distinct set of technical trade-offs.

The primary downsides to consider before using two BLE controllers simultaneously include:

1. Cumulative Jitter (Timing Variations)
While the baseline latency of a single BLE MIDI controller is generally quite low (around 7 to 12 milliseconds), adding a second BLE controller forces your computer’s single internal Bluetooth antenna to rapidly split its attention. Bluetooth transmits data packets in "connection intervals." When managing multiple active devices, these intervals can fluctuate. This creates jitter—meaning one chord might have a 7ms delay, while the next has a 15ms delay. For live-coding algorithmic music or tweaking filter sweeps, it is perfectly fine, but for recording precisely timed, rapid drum rhythms, the slight inconsistency can feel frustrating.  
CME

2. Bandwidth Throttling from Heavy Knob Automation
Musicians frequently run into a bandwidth bottleneck when turning multiple knobs simultaneously on BLE devices. A single note-on command uses very little data. However, turning a physical dial or slider sweeps through values 0 to 127, sending a dense flood of Continuous Controller (CC) messages. If you are twisting a knob on Controller A while simultaneously playing a melody on Controller B, the heavy stream of CC data can saturate the BLE connection, causing dropped messages, lagging parameters, or briefly stuck notes.

3. Local 2.4 GHz Signal Interference
Bluetooth operates on the highly crowded 2.4 GHz wireless frequency spectrum. If your performance space has an active Wi-Fi router, wireless computer mice, or active smart home devices nearby, the risk of intermittent packet loss spikes significantly. When a single BLE device drops a packet, it might result in a tiny lag spike; if you have two devices competing for the same congested airspace, it can result in temporary device disconnections.

The Verdict: Stick to the "Split" Protocol Setup
This is precisely why pairing one BLE controller with one 2.4 GHz USB dongle controller is the superior architecture. By routing one controller through your computer’s internal Bluetooth chip and the other through a dedicated, proprietary 2.4 GHz USB receiver (like the Roland WM-1D system), you completely separate their data traffic. They will not compete for bandwidth, your computer's Bluetooth antenna won't suffer from multi-device jitter, and you get rock-solid, completely independent control over your Strudel tracks.

You have hit a classic Strudel compilation trap! The reason you are getting that weird [mini] parse error at line 57 error is not because of line 57 of your actual script.

Instead, it is happening on line 126 because of how you called chord():

JavaScript
_$: chord(stepcat(...chordArray)) // <--- The culprit!
Why it crashed
In Strudel, chord() as a standalone function expects a plain string of a chord name (like "Cm7"). When you pass stepcat(...chordArray) into it, you are handing it a fully structural, complex JavaScript Pattern Object.

Because chord() doesn't native know what to do with a raw object as an argument, it implicitly triggers a .toString() conversion. This serializes your pattern object into a massive, 50+ line internal text string representation containing bracket arrays and configuration parameters. Strudel's mini-notation parser then tries to read this massive string as if it were a musical pattern. When it reaches line 57 of that internal dump, it encounters a double quote (") used in the serialization syntax, and it completely chokes.

How to Fix It
Change chord(stepcat(...chordArray)) to use method chaining instead: stepcat(...chordArray).chord(). This applies the chord structure cleanly to the pattern without forcing any string conversion.

In bassFromChords, you don't need to append an empty string (note + ""). Strudel's stepcat handles raw numbers perfectly well. Passing numbers avoids unexpected type-coercion issues down the line.

For consistency and to avoid errors down the road, your melodyFromChords function has been refactored to use the exact same [duration, value] array architecture as the bass. Passing pre-baked pure() pattern objects into seq() can trigger similar parser warnings; swapping it out for a clean stepcat(...melodyArray) makes your whole track ecosystem bulletproof.

    // Generate a single string of notes, e.g., "0 0 2 0 4 0..."

    for (const step of chordInfo) {

            const note = tones[Math.floor(Math.random() * tones.length)];
            // Optionally add an octave jump to make it more "melodic"
            const octave = Math.random() < 0.3 ? 7 : 0;

        }
    }
    return a;
}

function melodyFollowsChords(chordInfo, subdivisionsPerStep) {
    let a = [];
    for (const step of chordInfo) {
        // 1. Check if chord exists AND exists in your lookup table
        if (!step.chord || !chordTones[step.chord]) {
            // If no chord, push silence for the duration of the step
            a.push(silence.legato(step.dur));
            continue;
        }

        const tones = chordTones[step.chord];
        const subDur = step.dur / subdivisionsPerStep;

        for (let i = 0; i < subdivisionsPerStep; i++) {
            // 2. Safely pick a tone
            const note = tones[Math.floor(Math.random() * tones.length)];
            const octave = Math.random() < 0.3 ? 7 : 0;
            
            a.push(pure(note + octave).legato(subDur));
        }
    }
    return a;
}
Why this fixes the crash
The if (!step.chord || !chordTones[step.chord]) block acts as a "guard clause."

If step.chord is null (or undefined), it enters the if block, pushes silence, and uses continue to skip the rest of the loop for that step.

This prevents the code from ever trying to access chordTones[null].length.
